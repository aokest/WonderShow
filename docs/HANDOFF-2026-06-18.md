## 2. 立项背景

灵演最初不是一个单纯的遥控翻页工具，而是一个面向演讲、课程录制和活动表达的 macOS 演示导演台。最早的产品设想包含三条主线：

1. 演讲隔空控制：演讲者可以通过手势控制 PPT/Keynote/网页演示的翻页、放缩、标注等动作，让演讲更立体，减少被键盘和翻页笔束缚。
2. 演讲录制与后期合成：演讲时同时录制讲者和 PPT/屏幕，最终在时间轴上合成讲者全身、讲者特写、讲者画中画、PPT 高清全屏等多视角演讲视频。
3. 培训录屏课：在没有现场演讲时，也可以用电脑摄像头、外接摄像机、口袋相机、手机等输入源，同时录屏和录讲者，合成讲者视图、讲者画中画、纯 PPT 视图等课程视频。

本阶段的开发顺序经过调整：先把第 2、3 点，也就是“屏幕/窗口录制 + 讲者录制 + 项目保存 + 预览 + 导出”这条主链路打通；手势识别继续保留并改进，但暂时不让手势功能阻塞录制工作室能力。

## 3. 开发意图

本阶段的开发意图有四个：

- 把灵演从“手势导演台原型”推进到“可录、可存、可预览、可导出”的录制工作室雏形。
- 保持现有暖黑金 UI 语言，不大拆 appui 框架，只在必须的位置补交互和状态。
- 所有可见预览、最终预览、导出视频尽量使用同一套构图数据，避免监视器里看见的画中画和导出结果不一致。
- 以非破坏式工程为方向：原始讲者轨、屏幕轨、麦克风轨和 program 合成结果分开保存，未来可以继续做时间轴编辑。

一个重要产品原则：当前已经测试成功的录制、选源、画中画、预览、导出、收音能力，后续迭代不要随意重构。若必须改动，先补测试，再做小步替换。

## 7. 技术预研结论

### 手势识别路线：视觉优先

目前建议继续走视觉识别主线，而不是立刻转向外部硬件。

视觉识别优点：

- 不需要额外硬件，用户只用内置摄像头或外接摄像头即可开始。
- 能直接识别手部形状、手的位置、动作轨迹和身体上下文。
- 适合演讲场景中的“动态动作”，例如挥动翻页、双手放缩、停留确认。
- 训练数据可以通过灵演内置采样工具持续积累。
- 可用 MediaPipe/Vision 打底，再用自有模型做产品化微调。

视觉识别缺点：

- 受距离、光线、遮挡、背景、手部大小影响。
- 几米外识别静态手型会变难，动态动作必须依赖时间序列而不是单帧分类。
- 需要真实用户数据和回放测试集，否则容易陷入反复调参。

### 结构光/ToF/激光点阵

优点：

- 深度信息对分割手和身体更有帮助。
- 对背景干扰更稳，近距离识别精度较高。

限制：

- 外部硬件成本、连接、供电、眼安全、兼容性和售后复杂度明显增加。
- iPhone/iPad 的 TrueDepth/LiDAR 并不天然等价于一个可直接接入 Mac app 的通用传感器。
- 演讲场景往往需要几米距离和宽视角，消费级结构光/ToF 未必覆盖。

结论：可以作为未来高端硬件套装或增强输入，但不适合作为当前主线。

### 超声波

优点：

- 成本低，隐私压力小。
- 可做接近、挥动、粗略方向等低维交互。

限制：

- 空间分辨率不足，很难表达复杂手型。
- 对多人、环境反射、距离变化敏感。
- 不适合灵演当前要做的“演讲动作 + 手型 + 动态轨迹”。

结论：不建议作为主识别路线。

### 强化学习与动态手势

用户希望未来通过更好的强化学习框架提升远距离动态手势识别。这里建议分阶段处理：

1. 第一阶段不要直接用强化学习做原始识别。当前问题本质更像“时间序列分类 + 个性化校准 + 置信度策略”，应优先用监督学习、时序模型和主动学习。
2. 建立真实采样数据集：不同距离、光照、衣着、背景、摄像头、高矮站姿、左右手、快慢动作、误触动作都要覆盖。
3. 使用时序模型：TCN、轻量 Transformer、LSTM/GRU 或基于 landmarks 的 Temporal CNN，输入为连续多帧的 21 点手部几何、速度、加速度、双手距离变化。
4. 加入 hard negative mining：把误触最多的动作，如自然摆手、扶眼镜、拿遥控器、指屏幕但不翻页，专门作为负样本。
5. 加入在线个性化：用户做少量校准后，用相似度模板或小模型 adapter 调整阈值。
6. 强化学习更适合后续调“策略层”：例如不同用户/场地的阈值、冷却、确认窗口、动作触发策略，而不是第一步替代视觉检测。

推荐路线：视觉识别 + 真实数据采样 + 时序动态模型 + 主动学习/个性化校准。强化学习作为策略优化的后续增强。

## 8. Debug 记录与踩坑

### 屏幕录制权限看似已开，但源列表为空

现象：系统设置里已经允许灵演录屏，但 App 里扫描不到窗口或屏幕。

原因：macOS Screen Recording 权限经常需要 app 重启，且未授权时 `SCShareableContent` 可能直接拿不到可录制源。

处理：增加权限检测、提示、重新扫描；用户重新授权并重新打开 app 后恢复。

后续注意：若 bundle id 或签名变化，macOS 可能把它当成另一个 app，需要重新授权。

### 录制源列表抓出大量后台项目

现象：像会议软件共享屏幕一样选源时，列表里出现 Dock、Wallpaper、后台服务等无意义项。

原因：ScreenCaptureKit 默认能暴露较多系统 surface，不等于用户想录制的“可见活动窗口”。

处理：选源弹窗改用 on-screen windows 策略，过滤无标题、系统、后台、尺寸过小或不可录制项，同时保留屏幕项。

### PPT/Keynote 画面没有进入监视器和导出

现象：摄像头可录，但导出没有 PPT/屏幕画面。

原因：早期只是 UI 上有“目标应用/录制源”概念，真实 ScreenCaptureKit 选择和 raw track 写入没有完全接上；后续又遇到权限、窗口筛选、静态画面不推帧等问题。

处理：接入真实 `ScreenArchiveRecorder`，并让 `ScreenPreviewService` 和 writer 使用同一选择源；静态窗口通过 frame pump 持续写最新帧，避免没有新帧时生成空视频。

### 预览合成无反应或闪退

现象：“预览合成”点击后无响应，或在没有 program 文件时闪退/失败。

原因：预览逻辑假设 `Exports/program.mp4` 已经存在，且没有把 raw 轨缺失、空文件、无视频轨道等错误反馈给 UI。

处理：预览前调用渲染器尝试重合成；`ProgramVideoRenderer.validatePlayableVideo` 检查文件存在、非空、时长和视频轨；失败时显示明确原因。

### 导出视频点击后没有文件

现象：导出按钮可点击，但用户找不到生成文件。

原因：早期导出任务缺少进度状态和成功/失败反馈，部分失败被吞掉；也缺少输出文件校验。

处理：导出加进度 sheet、成功 alert、Finder 入口和输出校验；编码参数进入真实 writer 设置，并由测试覆盖。

### 画中画在监视器和导出中位置不一致

现象：监视器中拖好的画中画位置、大小、形状，预览/导出后对不上。

原因：UI 使用像素/视图坐标，渲染器使用视频归一化坐标；早期没有把 PiP keyframes 统一写入 manifest。

处理：将画中画位置、尺寸、形状转换成 `ProgramPictureInPictureGeometry` 和 keyframes；预览、导出、manifest 使用同一套数据。

### 录制中切换源后监视器不变

现象：未录制时切源有效，录制中切源后背景不跟着变。

原因：录制中的 `SCStream` 和预览状态没有被动态替换，writer 仍持有旧源。

处理：新增/接通 `updateSource`，切源时重建/更新 screen capture filter，并同步监视器预览和写盘轨。

### 屏幕画面模糊

现象：监视器里的屏幕画面不是高清。

原因：窗口尺寸、Retina scale、stream configuration 和缩略图/预览路径混用时，容易按逻辑点而不是像素写入。

处理：新增 `capturePixelSize`，按显示 scale 计算偶数像素尺寸；测试覆盖 Retina scale。

### 停止录制后时间和状态没清零

现象：停止后仍显示等待但时间停在上一段。

原因：UI 状态和真实录制生命周期没有完全统一。

处理：录制控制状态拆成 waiting/recording/paused，终止后弹窗选择保存或放弃，并清理状态、计时和录制中标记。

### 音频输入选择缺失

现象：只有视频输入设备，没有声音输入源选择。

原因：早期使用更简单的音频录制路径，没有暴露设备选择。

处理：`MicrophoneArchiveRecorder` 改用 `AVCaptureAudioFileOutput`，枚举麦克风/外接输入，支持暂停/继续。

### UI 低对比度

现象：导出设置、下拉标签、滑竿、输入设备菜单在深色背景下看不清。

原因：macOS 原生 Picker/Menu 在自定义深色面板中默认样式不可控。

处理：对关键选项使用自绘深色控件、金色高亮和统一按钮样式。

### 版本号不变

现象：app 迭代了，但 UI/Finder 里版本看起来没变。

原因：早期版本号写死或未随打包更新。

处理：`scripts/build-app.sh` 自动写入 `CFBundleShortVersionString = 0.7.YYYYMMDD` 和 `CFBundleVersion = YYYYMMDDHHMM`；Dashboard 从 bundle 读取版本。

## 9. 当前未解决和下一步计划

### P0：监视器中活动窗口抖动

用户观察：把窗口切到监视器中后，活动窗口画面在监视器里抖动，不稳定。

初步判断：

- 可能是捕获窗口 frame 或 contentRect 在活动窗口刷新/阴影/标题栏变化时来回变。
- 也可能是窗口源和显示源之间的尺寸换算、Retina scale、aspect-fit 布局每帧重算导致轻微跳动。
- 若录制中切源，`SCContentFilter` 更新和 preview frame 更新也可能出现短暂尺寸不一致。

建议修复方向：

1. 在 `ScreenPreviewService` 中记录同一 source 的稳定 capture rect，避免每帧用浮动 frame 重排。
2. 对窗口尺寸变化做 debounce，只有变化超过阈值才更新布局。
3. preview 层固定 aspect ratio，只更新图像内容，不让容器跟随每帧变化。
4. 增加真实窗口抖动复现日志：source id、frame、contentRect、pixelSize、preview image size。

### P1：美颜、调亮、镜像翻转

目标：提高讲者视图质量，尤其是课程录制和演讲画中画。

建议实现顺序：

1. 摄像头镜像翻转：先做 preview 与导出一致的 horizontal flip 开关。
2. 亮度/对比度/曝光补偿：用 Core Image filter 或 Metal pipeline，在 preview 和 raw/export 路径中统一使用。
3. 轻量美颜：先做肤色区域柔化、降噪、锐化、色温，不要一开始做重度人脸形变。
4. 将参数写入 project manifest，保证重新打开项目和重新导出可复现。

### P1：底部时间轴从占位变成真实编辑器

当前底部时间轴基本还是状态展示，不是真正可编辑的时间轴。

期望能力：

- 实时展示正在写入的屏幕轨、讲者轨、麦克风轨和合成轨。
- 支持轨道折叠/展开。
- 支持删除片段。
- 支持拖拽定位播放头。
- 支持多选片段。
- 支持只导出选中片段、选中轨道或选中素材。
- 支持选择某一个时间片段导出。
- 支持选择多段不连续时间片段导出，并按用户选择合成为一个新视频或导出为多个片段。
- 支持对一段或多段时间轴范围进行预览、裁剪、删除和重新合成。
- 支持展示音频波形、视频缩略图、画中画 keyframe、源切换事件、手势事件。

建议技术方向：

- 先在 manifest 中明确 `TrackSegment`、`TimelineEvent`、`PiPKeyframe`、`SourceSwitchEvent`、`TimelineSelection` 和 `ExportRange`。
- UI 层先做只读真实时间轴，再做编辑。
- 所有编辑保持非破坏式，不直接改 raw 文件。

### P1：macOS 顶部常驻状态栏、桌面悬浮窗与迷你工具条

目标：让灵演在录制或演讲时不占据大屏幕空间。

期望能力：

- App 成为 macOS 菜单栏常驻小标签。
- 双击菜单栏图标可以收起/展开主界面。
- 右键可以打开常用配置：开始/暂停/继续/结束录制、选择源、音频输入、画中画设置、打开项目等。
- 录制过程中主界面可缩成一个可在整个桌面自由拖拽的 mini toolbar。
- mini toolbar 保留暂停、继续、结束录制、录制时间。
- mini toolbar 还需要提供“切换活动窗格/录制源”的按钮，让演讲者在不展开主界面的情况下切换正在录制或监视的窗口。
- mini toolbar 需要避免遮挡被录制内容，后续应支持吸边、透明度、锁定位置和录制时隐藏自身等策略。

建议技术方向：

- 使用 `NSStatusItem` + SwiftUI popover/menu。
- 使用独立 `NSPanel` 或 borderless floating window 承载桌面悬浮 mini toolbar。
- 主窗口管理从 SwiftUI view 内状态迁移到 app coordinator。
- 录制控制状态应成为全局 observable model，菜单栏、迷你 toolbar、主 Dashboard 共用同一状态源。
- “切换活动窗格”应复用当前 ScreenCaptureKit 源选择/切源能力，避免做第二套源状态。

### P1：动态手势识别准确度

目标：几米外也能准确识别动作，而不是只识别近距离静态手型。

建议计划：

1. 完善采样工具：把距离、光照、摄像头、动作类型、成功/失败标签写入样本元数据。
2. 建立回放测试集：每次改识别算法都跑相同视频，避免靠现场感觉调参。
3. 引入时序模型：使用连续 landmarks、速度、加速度、双手距离变化识别动态动作。
4. 加入 hard negatives：自然摆手、拿东西、指屏幕、扶眼镜等都要作为“不要触发”样本。
5. 加入用户校准：让每个用户少量录几个动作，生成个性化阈值或小 adapter。
6. 强化学习放在策略层：优化触发阈值、冷却和确认策略，不作为第一版核心识别器。

### P2：录制中音频输入源切换

用户希望未来可以在录制过程中切换音频输入，且不影响收音和时间轴。

建议做法：

- 不要直接覆盖同一音频文件。
- 每次切换生成新的 audio segment，并在 manifest 中记录 source id、start/end、offset。
- 合成时按时间轴拼接或混音。
- UI 上显示音频源切换事件和当前活跃麦克风。

### P2：多机位时间轴

未来希望多个视频输入源可以同时勾选，并作为多轨录制。

建议：

- 录制模型已经能表达多摄像头和多麦克风输入，但 App UI/真实采集还未完全支持多路同时写盘。
- 先支持两个摄像头并行：主讲者 + 侧拍/全身。
- 每个机位独立 raw track，program timeline 只记录当前使用哪一个机位和构图。

### P2：授权验证、付费激活与商业化分发

目标：未来灵演需要能上架 App Store，或通过互联网直接销售，并让用户可以自助付费、主动联系付费或在软件内付费后激活。

需要设计的问题：

- App Store 路线：优先研究 StoreKit 2、订阅/一次性购买/内购解锁、恢复购买、收据验证和家庭共享策略。
- 直营网售路线：需要设计账号体系、订单系统、许可证服务器、离线宽限期、设备绑定、换机恢复和撤销机制。
- 主动联系付费路线：软件内可提供“联系购买 / 获取企业授权 / 输入授权码”的入口，适合早期私域销售和企业客户。
- 免费试用策略：可考虑限制录制时长、导出水印、导出分辨率、项目数量或高级功能，而不是限制基础体验。
- 激活状态需要写入本地安全存储，并支持重新安装/换机后的恢复。
- 不要把密钥、私钥、签名盐或支付凭证写入仓库或客户端明文。
- 对外发布前需要单独形成商业化与授权设计文档，明确 App Store 与直营版是否共用同一套授权模型。

建议技术方向：

- App Store 版优先走 StoreKit 2 和 Apple receipt/server API。
- 直营版优先做签名许可证文件 + 服务器校验 + 离线 grace period。
- 客户端只保存短期 token 或签名许可证，不保存可伪造的“已付费”布尔值。
- 授权逻辑不要散落在 UI 里，应集中为 `LicensingService` / `EntitlementStore`。

### P2：多端点与跨平台支持

目标：未来灵演不只运行在 macOS，还可能支持 iPad、iPhone、Android、PC，甚至鸿蒙。

产品分层建议：

- macOS：继续作为完整录制工作室主端，负责屏幕/窗口捕获、项目管理、合成和导出。
- iPad/iPhone：优先作为遥控器、补充摄像头、提词器、手势采样端或移动导播端，再评估是否做完整编辑器。
- Android：优先作为补充摄像头/遥控器/采样端，降低全量录制工作室的首版复杂度。
- Windows PC：可作为第二个完整桌面录制端，但需要单独处理窗口捕获、音频环回、编码器和驱动差异。
- 鸿蒙：先作为长期兼容方向记录，待核心协议和商业验证稳定后再评估。

技术建议：

- 把项目格式、时间轴 manifest、导出意图、手势事件协议设计为平台无关。
- 把 macOS 专属能力如 ScreenCaptureKit、AVFoundation、NSStatusItem 隔离在平台层。
- 移动端和桌面端之间可以先通过局域网协议传输摄像头画面、控制命令、手势事件和录制状态。
- 跨端前先稳定 `.wondershow` 项目格式，避免每个平台各自保存一套不可互通的数据。

### P2：多皮肤与主题系统

目标：除了当前暖黑金暗色皮肤，未来支持更多视觉风格，让不同场景选择不同 UI 气质。

候选皮肤：

- 暖黑金：当前默认，适合演讲导演台和舞台感。
- 极简浅色：白色/浅灰背景，适合教育、办公和长时间剪辑。
- 商务版：更克制的深灰/蓝灰/中性色，适合企业培训和会议场景。
- 极客版：高对比深色、代码/仪表盘感，适合开发者和调试场景。
- 金黄版：保留品牌金色但降低暗色面积，适合品牌展示。
- 黑色/白色视图切换：至少应支持系统级 light/dark 两套基础主题。

技术建议：

- 先抽象 `DesignToken`：颜色、字体、间距、圆角、阴影、边框、按钮状态和图表颜色。
- 不要在 SwiftUI view 内散落硬编码颜色；现有 `ConsolePalette` 后续可演进为主题 provider。
- 主题切换要覆盖主界面、弹窗、源选择器、导出进度、时间轴、mini toolbar 和未来授权弹窗。
- 新主题必须经过截图检查，避免低对比度、文字不可见和按钮状态不明显的问题。

## 11. 后续迭代注意事项

- 不要把录制源选择、画中画、预览、导出和音频选择当成 UI 假按钮处理；现在这些已经接入真实功能。
- 任何涉及画中画坐标的改动，都要同时验证监视器、manifest、预览、导出四处一致。
- 任何涉及 ScreenCaptureKit 的改动，都要测试权限未开、权限刚开、重启后、录制中切源、静态窗口、Retina 屏幕。
- 任何涉及导出的改动，都要验证真实输出文件的分辨率、帧率、编码和可播放性。
- 任何涉及录制状态的改动，都要验证等待、录制中、暂停、继续、终止保存、终止放弃。
- 文档更新入口是 `docs/INDEX.md` 和 `docs/DOCLOG.md`，阶段性总结继续放到 `docs/HANDOFF-*.md`。
